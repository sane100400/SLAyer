03 — Solution

SLA 룰 정의 → 게이트 실행.
세 명령, 끝.

Service Level Agreement — 팀이 정한 보안 규칙을 위반하면 SLAyer가 배포를 차단(slay)한다.

① Contaminated
slayer start . → 배포 차단
② Burning
slayer patch . → AI CLI 최소 수정
③ Cleared
전 룰 통과 → 🚀 Deployment Approved

04 — Engineering

데이터 수집부터 패치까지
전부 직접 설계.

①
Data
Pipeline
GitHub CLAUDE.md 레포 직접 수집 · AST 빈도 분석
중요도 5축(공격 가능성·피해도·TTE·탐지신뢰도·AI증폭) × 0.6 + log(빈도) × 0.4 → 상위 7종 확정
②
Deterministic
Detection
Python: ast 모듈 완전 파싱 — AST node 순회 (AI 없음, 100% 결정적)
JS/TS: 컨텍스트 인식 regex — SSRF는 URL 인자만, NO_EXEC는 객체 메서드 제외 (db.exec FP 방어)
③
AI CLI
Delegation
anthropic SDK 없음 — 로컬 설치된 claude/codex/gemini subprocess 위임
AI 응답에서 코드블록 추출 · 시크릿 마스킹 후 프롬프트 전송 (유출 방지)
④
Patch
Quality
취약점 1개씩 패치 → 재스캔 → 반복 (최대 20회 루프)
매 패치마다 ast.parse / node --check / tsc --noEmit 문법 검증 · 실패 시 원본 복원 · symlink 경계 방어
