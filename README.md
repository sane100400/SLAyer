# SLAyer

> **Security scanner for AI-generated web services**  
> CMUX × AIM Hackathon 2026 · Developer Tooling Track

Detects 7 security vulnerability patterns in web service code (Python · JS · TS) generated by AI coding tools (Claude / GPT / Cursor), then patches them automatically using your locally installed AI CLI and opens the deployment gate.

## Quickstart

```bash
pip install -e ".[dev]"

tmp="$(mktemp -d)"
cp dataset/slayer-bench-v0/vulnerable/python/py_secret_exec_sql.py "$tmp/app.py"

slayer start "$tmp/app.py"   # scan → violation list
slayer patch "$tmp/app.py"   # scan → auto-patch → 🚀 Deployment Approved
slayer start "$tmp/app.py"   # rescan → clean
```

1. `slayer start` detects security issues without any API key or config.
2. `slayer patch` calls your locally installed Claude Code / Codex / Gemini CLI to fix them.
3. SLAyer validates the patch syntax, rescans, and confirms deployment readiness.

---

No API key required. Patching works if Claude Code / Codex / Gemini is already installed locally.  
`slayer start` works without any AI CLI. `slayer patch` requires one AI CLI installed.

## Why SLAyer?

Existing tools like Bandit and Semgrep use general-purpose security rules.  
SLAyer targets **recurring vulnerability patterns specific to AI-generated code**, identified from real data.

| | SLAyer | Bandit | Semgrep |
|---|---|---|---|
| Target | AI-generated code patterns | Python general | General (requires rule files) |
| Languages | Python · JS · TS | Python | Many |
| Auto-patch | Yes, via AI CLI | No | No |
| Config | Zero — runs immediately | Required | Rule file required |

| Root cause | Pattern |
|------|------|
| "Make it work first" prompts | Hardcoded credentials, external calls, `shell=True` |
| Outdated training data from tutorials | f-string SQL, `Math.random()` for tokens |
| Dev examples shipped to production | `DEBUG=True`, `debug: true` |
| "Remove the error" prompts | `except: pass`, empty `catch {}` |

## Why these 7 rules?

We collected and analyzed GitHub repositories that contain `CLAUDE.md` files and measured pattern frequency,  
then selected the final 7 using a **frequency × importance composite score**.

```
score = importance_weighted_sum × 0.6 + log10(frequency) / log10(max) × 5 × 0.4
```

Importance is evaluated on 5 axes: exploitability (25%) · impact severity (25%) · time-to-exploit (20%) · detection confidence (15%) · AI amplification factor (15%).

| Rule | Score | Severity |
|----|-----------|----------|
| NO_HARDCODED_SECRETS | 5.00 | critical |
| NO_EXEC | 3.98 | critical |
| SQL_PARAM_BINDING | 3.91 | high |
| NO_DEBUG_MODE | 3.87 | high |
| NO_NETWORK | 2.63 | critical |
| NO_WEAK_RANDOM | 2.91 | high |
| NO_BARE_EXCEPT | 3.00 | medium |

> Full methodology: [`spec.md § 0.55`](./docs/spec.md)

## Supported Languages

Python (`.py`) · JavaScript (`.js`, `.jsx`) · TypeScript (`.ts`, `.tsx`)

## Supported OS and AI CLI

- **OS:** Windows, macOS, Linux
- **Python:** 3.11+
- **AI CLI patching:** Claude Code, Codex CLI, or Gemini CLI installed in local PATH

```bash
slayer model auto      # auto-detect: claude → codex → gemini
slayer model claude    # lock to Claude Code, save to .slayer.yml
slayer model codex     # lock to Codex CLI
slayer model gemini    # lock to Gemini CLI
```

## Usage

### slayer start — scan

```bash
slayer start .                   # scan current directory
slayer start dataset/slayer-bench-v0/vulnerable/python/py_secret_exec_sql.py
slayer start . --format json     # JSON output for CI/CD
```

Example output:
```
  ● CRITICAL  NO_HARDCODED_SECRETS  py_secret_exec_sql.py:3
              API_KEY = "sk-prod-abc123..."
              → Use os.environ.get('API_KEY') — keep secrets out of the code.

  ▲ HIGH      SQL_PARAM_BINDING     py_secret_exec_sql.py:7
              cursor.execute(f"SELECT * FROM users WHERE name = '{query}'")
              → Use cursor.execute('SELECT ... WHERE name=?', (name,))

  3 violations · 🔒 Deployment BLOCKED

  Run slayer patch <path> to fix automatically.
```

### slayer patch — auto-patch

```bash
slayer patch "$tmp/app.py"       # scan → patch → rescan
slayer patch . --format json     # JSON result output
```

Example output:
```
  Patching via claude...

  ✓  app.py patched

  Patch explanations:
  • NO_HARDCODED_SECRETS  app.py:3  — Moved secret values out of the code
    Replaced hardcoded keys or passwords with environment variable lookups...

  🚀 Deployment Approved
```

### slayer model — AI CLI configuration

```bash
slayer model              # show detection status and current setting
slayer model claude       # save claude to .slayer.yml
slayer model codex        # save codex
slayer model auto         # reset to auto-detect
```

## Exit Codes

| Code | Meaning |
|------|------|
| `0` | No violations — Deployment Approved |
| `1` | Violations found — Deployment BLOCKED |
| `2` | Execution error |

## CI/CD Integration

```bash
# Check deployable via JSON
slayer patch . --format json | jq -e '.deployable'
```

  - repo: local

## Benchmark

### slayer-bench-v0 — hand-curated

32 cases (22 vulnerable + 6 FP-free + 4 blind_spot), all executed as a smoke benchmark:

**In-scope detection (direct patterns):** all vulnerable cases are blocked, and all fixed/FP-free cases are approved in the pytest suite. Known blind spots are documented below.

| Metric | Value |
|------|-----|
| TP (true positive) | 22 |
| FP (false positive) | 0 |
| TN (true negative) | 6 |
| FN (false negative) | 0 |

**Blind spot cases (known limitations — intentionally not detected):**

| Case | Rule | Reason |
|--------|-----|------|
| Variable-mediated SSRF | NO_NETWORK | URL assembled from parts, not direct user input at call site |
| Secret via function return | NO_HARDCODED_SECRETS | `return "sk-..."` form — only direct assignment patterns are detected |
| split() shell injection | NO_EXEC | `shell=False` + `cmd.split()` — passes shell check but still injectable |
| Array-join SQL | SQL_PARAM_BINDING | `parts.join(' ')` — not a template literal or simple concat pattern |

|----|--------|
| NO_WEAK_RANDOM | 5 |

| NO_HARDCODED_SECRETS | 3 |
| SQL_PARAM_BINDING | 3 |

## Limitations

- **Indirect SSRF not detected**: Only detects `requests.get(user_input)` form directly. Patterns where the URL is stored in a variable first (`url = user_input; requests.get(url)`) are not currently detected.
- **Dynamic code not detected**: Patterns that dynamically assemble strings into `eval()` or `exec()` are outside detection scope.
- **Security context disambiguation**: `NO_WEAK_RANDOM` detects `random` usage near security context keywords (token, session, OTP), but does not identify every possible security context.
- **Patch scope**: AI patching only fixes detected violations. Undetected vulnerabilities remain.

## Architecture

```
slayer/
├── cli.py              # slayer start / patch / model
├── analyzers/
│   ├── py_analyzer.py  # Python AST-based detection (no AI required)
│   └── js_analyzer.py  # JS/TS regex-based detection (no AI required)
├── ai_runner.py        # AI CLI detection (claude→codex→gemini) + delegation
├── patcher/
│   └── llm_patcher.py  # SLAyer-scoped auto-patch
└── models.py           # Pydantic data models
```

Detection runs deterministically without AI. The AI CLI is used for patching only.
